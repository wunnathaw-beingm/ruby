& နဲ့ to_proc Method

& နဲ့ပတ်သက်ပြီး အရင်က ရှင်းပြခဲ့သလိုပါပဲ Proc object ကို block အနေနဲ့ ပေးလိုက်တဲ့အချိန်မှာparameter name ရဲ့ ရှေ့မှာ & ထည့်ပေးဖို့လိုအပ်ပါတယ်။

ဘာဖြစ်လို့ & ကို ထည့်ပေးရတာလဲ။ & က ဘာလုပ်ပေးတာလဲ ဆိုတော့ & ရဲ့အလုပ်က Proc object ကို block အနေနဲ့သုံးလို့ရအောင်လုပ်ပေးတာပါပဲ။ & လုပ်ပေးတာကို Semantics အရပြောရမယ်ဆိုရင်

  1. & ရဲ့ ညာဘက်မှာရှိတဲ့ object (sample_proc) ကို to_proc method ကိုခေါ်လိုက်ပါတယ်။(sample_proc.to_proc)
  2. ‌ အဲ့တော့ return value အနေနဲ့ proc object ကို return ပြန်ပါမယ်။
  3. ရလာတဲ့ return value(proc object) ကို block သုံးမယ့် method ကိုပေးလိုက်ပါတယ်။ 
sample_proc = Proc.new { |s| s.reverse }
['apple', 'orange', 'banana'].map(&sample_proc) 
#=>  ["elppa", "egnaro", "ananab"]

အဲ့ဒီတော့ အရင်ကတည်းက proc object ကိုပေးလိုက်တယ်ဆိုရင် ဘာဖြစ်မလဲ။& က to_proc method ကိုခေါ်မှာပါပဲ။ ဒါပေမဲ့ return value ကလက်ရှိ proc object ကိုပဲ return ပြန်တာပဲဖြစ်မှာပါ။
အောက်က example ကို ကြည့်ပါ။ proc object ကို to_proc method ခေါ်လိုက်လဲ အရင် proc object နဲ့ return value က တူတူပဲဖြစ်ပါတယ်။ 

sample_proc = Proc.new { |s| s.reverse }
other_proc = sample_proc.to_proc
other_proc.equal?(sample_proc)   #=> true

ဒါပေမဲ့လို့ Ruby မှာ Proc object မဟုတ်တဲ့ တခြား object တွေထဲမှာ to_proc method ကို ခေါ်သုံးလို့ရတဲ့ဟာတွေရှိကြပါတယ်။အဲ့ထဲက တစ်ခုကတော့ symbol ပါ။ဥပမာ အောက်ပါအတိုင်း :split ဆိုတဲ့ symbol ကို to_proc method သုံးပြီး Proc object အဖြစ် ပြောင်းပါမယ်။ 

split_proc = :split.to_proc
split_proc
#=>  #<Proc:0x0000000003665398(&:split)>

ရလာတဲ့ proc object ကို parameter တစ်ခုတည်းပေးပြီး run လိုက်ရင် first parameter ကိုပဲ receiver လုပ်၊ အဲ့ဒီ receiver ကပဲ အရင် symbol name နဲ့တူတဲ့ method ကို ခေါ်သုံးသွားမှာပါ။ 

split_proc = :split.to_proc
# parameter တစ်ခုတည်းဆိုရင် space နဲ့ split လုပ်
# ('a-b-c-d e').split နဲ့တူတူပဲ
split_proc.call('a-b-c-d e')   #=> ["a-b-c-d", "e"]

parameter 2 ခု ပေးမယ်ဆိုရင် ပထမ parameter က receiverဒုတိယ parameter က symbol ကနေ ခေါ်သုံးလိုက်တဲ့ method ရဲ့ 1st parameter ဖြစ်လာပါလိမ့်မယ်။ 

split_proc = :split.to_proc
# ('a-b-c-d e').split('-') နဲ့တူတူပဲ
split_proc.call('a-b-c-d e', '-')   #=> ["a", "b", "c", "d e"]

parameter 3 ခု ပေးမယ်ဆိုရင်3 ခုမြောက် parameter က symbol ကနေ ခေါ်သုံးလိုက်တဲ့ method ရဲ့ 2nd parameter ဖြစ်လာပါလိမ့်မယ်။ 

split_proc = :split.to_proc
# ('a-b-c-d e').split('-', 3) နဲ့တူတူပဲ
# 3 ဆိုတာက 3 ခုပဲ split လုပ်မယ်ဆိုပြီး ပြောလိုက်တာ
split_proc.call('a-b-c-d e', '-', 3)   #=> ["a", "b", "c-d e"]

အဲ့ဒီတော့ ဒီနေရာမှာ တစ်ခုကြည့်ကြည့်ရအောင်ပါ။
အောက်ပါ code 2 ခုဟာ output အတူတူပါပဲ။ 

['japan', 'us', 'myanmar'].map{ |s| s.upcase }
#=>  ["JAPAN", "US", "MYANMAR"]

# များသောအားဖြင့် အပေါ်က code လို ‌ရေးကြရတဲ့ဟာကို အောက်က အတိုင်း အတိုချုံ့ပြီးရေးလေ့ရှိကြပါတယ်။
['japan', 'us', 'myanmar'].map(&:upcase)
#=>  ["JAPAN", "US", "MYANMAR"]

အပေါ်မှာရှင်းပြထားခဲ့တော့ အောက်က code က ဘယ်လို run သွားလဲဆိုတာကို သဘောပေါက်လောက်ပြီထင်ပါတယ်။
semantics အရပြောရမယ်ဆိုရင်

  1. &:upcase က symbol ဖြစ်တဲ့ :upcase ကို to_proc method ကိုခေါ်ပေးလိုက်တယ်။(:upcase.to_proc)
  2. symbol :upcase က proc object ဖြစ်သွားပြီးတော့ map method parameter ထဲကို block အနေနဲ့ parameter passing လုပ်လိုက်တယ်။
  3. No.2 က proc object က map method ကနေပြီးတော့ array ထဲက data တွေကို first parameter အနေနဲ့ယူထားလိုက်တယ်။first parameter က upcase method ရဲ့ receiver ဆိုတော့ array ထဲက data တွေက upcase method ကို လှမ်းခေါ်ပြီး run မယ် ။
  4. map method က array အသစ်တစ်ခုဆောက်ပြီးတော့ Proc object ရဲ့ return value တွေကို array အသစ်ထဲမှာ သိမ်းထားလိုက်တယ်။
  5. 3 နဲ့ 4 ကို မပြီးမချင်း run ပေးလိုက်ပြီးတော့ နောက်ဆုံးထွက်လာတဲ့ array အသစ်ထဲက data တွေက upper case တွေဖြစ်၊ 
    ပြီးတော့ map method က return value အနေနဲ့ အဲ့ဒီ array အသစ်ကို return ပြန်ပေးလိုက်တယ်။

ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Rubyရဲ့‌ အခြေခံအချက်‌တွေ၊ Yield and Proc

Proc and Lambda

Proc Objectကို create လုပ်ပုံလုပ်နည်း 4 မျိုးရှိပါတယ်။
Proc.new ကိုသုံးပြီးတော့ Proc object ကို create လုပ်တာရယ်
proc method နဲ့ Proc object ကို create လုပ်ပုံလုပ်နည်းတွေက ရှင်းပြခဲ့ပြီးပါပြီ။
ဆက်လက်ပြီးတော့ အောက်ပါ syntax ပုံစံမျိုးတွေနဲ့လည်း Proc object ကို create လုပ်နိုင်ပါသေးတယ်။

# create Proc object with lambda symbol
proc_sample = ->(a, b) { a + b }

# create Proc object with lambda
proc_sample2 = lambda { |a, b| a + b }

အပေါ်က parameter မှာသုံးထားတဲ့ လက်သည်းကွင်း () ကို မထည့်လဲရပါတယ်. 

proc_sample = -> a,b { a + b }

parameter သာမရှိဘူးဆိုရင် အကုန်လုံးကို ချုံ့ရေးလို့ရပါတယ်။ 

proc_sample = -> { 'Hello!' }

{} အစား do…end လည်းသုံးနိုင်ပါတယ်. 

-> (a, b) do
  a + b
end

Proc.new နဲ့ အတူတူပဲ parameter မှာ default value ကို ပေးလို့ရပါတယ်။ 

proc_sample = ->(a = 0, b = 0) { a + b }

ပုံမှန် parameter ကိုခေါ်သုံးတဲ့နေရာမှာတော့ Proc.new ရော lambda ရောမှာ ကွာခြားတာမရှိပါဘူး။
အောက်မှာ Proc.new နဲ့ lambda ရဲ့ code ကို ပြထားပါတယ်။ 

#Proc.new နဲ့ create လုပ်တာ
proc_sample = Proc.new { |a, b| a + b }
proc_sample.call(1, 2)   #=> 3

# lambda နဲ့ create လုပ်တာ
proc_sample2 = ->(a, b) { a + b }
proc_sample2.call(1, 2)  #=> 3

ဒါပေမဲ့ Proc.new နဲ့ lambda သုံးတာမှာ ကွာခြားချက်လေးတွေ အများကြီးရှိပေမယ့် ဒီနေရာမှာ 2 ချက် ပြောပြချင်ပါတယ်။
ကွာခြားချက် ပထမတစ်ချက်အနေနဲ့ lambda က Proc.new ထက် parameter ကို check လုပ်ရာမှာ strict ဖြစ်စေပါတယ်။ 

#Proc.newမှာparameterကnilဖြစ်လဲerrorမတက်အောင်to_i method ကိုသုံးထားတာ
proc_sample = Proc.new { |a, b| a.to_i + b.to_i }
proc_sample.call(1)   #=> 1    #Procရဲ့parameter 1 ခုတည်းပေးလဲခေါ်နိုင်တယ်
proc_sample.call(1, 2, 3)   #=> 3　　#Procရဲ့parameter 3 ခုပေးလဲခေါ်နိုင်တယ်

#ラムダ
proc_sample2 = ->(a, b) { a.to_i + b.to_i }
proc_sample.call(1) # lambdaကျတော့ parameter မပြည့်စုံလို့ errorတက်မယ်
#=> ArgumentError: wrong number of arguments(given 1, expected 2)
proc_sample.call(1, 2, 3)
#=> ArgumentError: wrong number of arguments(given 3, expected 2)

Proc.new ရော lambda ရော Proc class ရဲ့ instance ဖြစ်ပါတယ်။Proc class ရဲ့ instance က Proc.new အနေနဲ့ create လုပ်ထားတာလား
lambda အနေနဲ့ create လုပ်ထားတာလားဆိုတာကို lambda? method ကိုသုံးပြီး ခွဲခြားနိုင်ပါတယ်။ lambda? method ပုံစံဖြစ်လို့ lambda အနေနဲ့ object ကို create လုပ်မယ်ဆိုရင် return value က true ကို ပြန်ပေးပါမယ်။ 

# Proc.new နဲ့ create လုပ်ထားတာဆိုရင်
proc_sample = Proc.new { |a, b| a + b }
proc_sample.lambda?    #=> false
proc_sample.class      #=> Proc

# lambda နဲ့ create လုပ်ထားတာဆိုရင်
proc_sample2 = ->(a, b) { a + b }
proc_sample2.lambda?  #=> true
proc_sample2.class    #=> Proc

■Proc.new နဲ့ lambda ရဲ့ returnနဲ့break ကွာခြားချက်

ဒုတိယကွာခြားချက်က Proc.newနဲ့ lambda အတွင်းထဲ return နဲ့ break ကို သုံးတဲ့အချိန်ကွာခြားတာဖြစ်ပါတယ်။ အောက်မှာ return ကို သုံးတဲ့ပုံစံကို ပြထားပါတယ်။ 

def proc_return
  proc_sample = Proc.new { |n| return n * 10 }
  result = [1, 2, 3].map(&proc_sample)
  "result: #{result}"
end

def lambda_return
  proc_sample = ->(n) { return n * 10 }
  result = [1, 2, 3].map(&proc_sample)
  "result: #{result}"
end

proc_return   #=> 10
lambda_return #=>  "result: [10, 20, 30]"

Proc.new မှာ return ကိုသုံးမယ်ဆို method ကြီးတစ်ခုလုံးထဲကနေ ထွက်သွားပါမယ်။အဲ့ဒါကြောင့် result က 10 ပဲထွက်လာပြီး ကျန်တာတွေက ဆက်မ run တော့တာပါ။lambda ဆိုရင်တော့ method ကြီးထဲကထွက်တာမဟုတ်ဘဲ lambda အတွင်းထဲက process ကနေပဲထွက်သွားတာပါ။ method ထဲကနေ ထွက်မသွားတဲ့အတွက် map loop ကို အဆုံးထိ run သွားပါတယ်။

အောက်ကတစ်ခုကတော့ break သုံးတဲ့ example ဖြစ်ပါတယ်။ 

def proc_break
  proc_sample = Proc.new { |n| break n * 10 }
  result = [1, 2, 3].map(&proc_sample)
  "result: #{result}"
end

def lambda_break
  proc_sample = ->(n) { break n * 10 }
  result = [1, 2, 3].map(&proc_sample)
  "result: #{result}"
end

proc_break   #=> LocalJumpError (break from proc-closure)
lambda_break #=>  "result: [10, 20, 30]"

Proc.new အတွင်းထဲမှာ break ကိုသုံးမယ်ဆို exception တက်ပါမယ်။
labmda မှာသုံးမယ်ဆိုရင်တော့ exception မတက်ဘဲနဲ့ လုပ်ရမယ့် process တွေကို အဆုံးထိ run ပေးသွားမှာဖြစ်ပါတယ်။ break ကိုသုံးပေမယ့် map method ထဲကနေမထွက်ဘဲ အဆုံးထိ run သွားပါတယ်။

ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Rubyရဲ့‌ အခြေခံအချက်‌တွေ၊ Yield and Proc

Proc

proc class ဟာ block ကို object အဖြစ် ပြောင်းဖို့အတွက် class တစ်ခု ဖြစ်ပါတယ်။
ဥပမာ string class ဆိုရင် string အတွက်။
integer class ဆိုရင် integer number တွေအတွက် ဖြစ်ပြီးတော့ proc class ကတော့ block ဒါမှမဟုတ် process တစ်ခုကို ရည်ညွှန်းပါတယ်။
Proc class instance (Proc object) တစ်ခုကို create လုပ်မယ်ဆိုရင်အောက်ပါအတိုင်း Proc.new မှာ block ကို ပေးလိုက်နိုင်ပါတယ်။

hello_proc = Proc.new do
    'Hello!'
end
#do ... end အစား {} ကို သုံးလဲ ရပါတယ်.
hello_proc = Proc.new{'Hello!'}

Proc.new ကို မသုံးဘဲ နောက်တစ်နည်းအနေနဲ့ Kernel module ရဲ့ proc method ကိုသုံးပြီးလည်း create လုပ်နိုင်ပါတယ်။

# Proc.new ကိုမသုံးဘဲ proc method ကိုသုံးတာ
add_proc = proc { |a, b| a + b }

Proc object ကို ဒီတိုင်းထားရုံနဲ့တော့ run မသွားနိုင်ပါဘူး။
run စေချင်ရင်တော့ call method ကို သုံးမှရပါမယ်။

hello_proc = Proc.new{'Hello!'}
hello_proc.call     #=> 'Hello!'

run တဲ့အချိန်မှာ parameter ပေးပြီးတော့ Proc object ကို run နိုင်ပါတယ်။

add_proc = Proc.new{ |a, b| a + b}
add_proc.call(10, 20)   #=> 30

parameter မှာ default value ကို ပေးနိုင်ပါတယ်။

add_proc = Proc.new{ |a = 0, b = 0| a + b}
add_proc.call           #=> 0
add_proc.call(10)       #=> 10
add_proc.call(10, 20)   #=> 30

Proc ကို run ဖို့အတွက် call method အပြင် တခြားနည်းတွေနဲ့လဲ ခေါ် run လို့ရပါတယ်။

add_proc = Proc.new { |a, b | a + b}

puts "use call method: #{add_proc.call(1, 2)}"
puts "use yield method: #{add_proc.yield(1, 2)}"
puts "use .() : #{add_proc.(1, 2)}"
puts "use [] : #{add_proc[1, 2]}"
puts "use === : #{add_proc === [1, 2]}"

#=> use call method: 3
#   use yield method: 3
#   use .() : 3
#   use [] : 3
#   use === : 3

■Proc Object ကို block အနေနဲ့ parameter လက်ခံတာ 

proc object ဟာ block ပုံစံမျိုးဖြစ်နေပေမယ့် block နဲ့ မတူတာက object အနေနဲ့လုပ်နိုင်တာပဲဖြစ်ပါတယ်။ ဘယ်လိုလဲဆိုတော့ variable ထဲ ထည့်ပြီးတော့  parameter passing ပေးနိုင်တာတွေ။
proc object ရဲ့ method call တွေကို ခေါ်နိုင်တာတွေဖြစ်ပါတယ်.

yield example မှာတုန်းက block ကို & သုံးပြီးတော့ parameter အနေနဲ့ လက်ခံနိုင်တယ်လို့ပြောခဲ့ပါတယ်။ အဲ့ဒီမှာ ထပ်ဖြည့်ပြီးပြောချင်တာက parameter မှာ ပေးခဲ့တဲ့ block ဆိုတာဟာ proc object ဖြစ်ပါတယ်။ ဒီလို output ထုတ်ကြည့်လို့ရပါတယ် 

def hoge(&block)
    puts block.class
    text = block.call('Kentaro')
    puts text
end

hoge do |text|
    "hello, #{text}"
end

#=> Proc
#   hello, Kentaro

■Proc Object ကို ပုံမှန် parameter အနေနဲ့လက်ခံတာ

Proc Object ကို အောက်ပါအတိုင်း & မထည့်ဘဲ ပုံမှန် parameter အနေနဲ့လက်ခံနိုင်ပါတယ်. 

def hoge(regular_proc)
    puts 'Hello!'
    text = regular_proc.call('How are you?')
    puts text
    puts 'Good morning!'
end

greeting_proc = Proc.new { | text | text * 2 }
hoge(greeting_proc)

#=> Hello!
#   How are you?How are you?
#   Good morning!

ဒီနေရာမှာ တစ်ခုပြောချင်တာတစ်ခုက block အနေနဲ့ parameter ပေးရင် parameter ကို အများဆုံး 1 ခုပဲ ပေးလို့ရပါတယ်.ဒါပေမယ့် Proc အနေနဲ့ parameter ပေးမယ်ဆိုရင်တော့ parameter မှာ ဘယ်နှခုပေးပေး ‌ပေးလို့ရပါတယ်.အောက်က ဥပမာမှာဆိုရင် parameter 3 ခုနဲ့ ပေးထားတာပါ. 

def hoge(proc_1, proc_2, proc_3)
    puts proc_1.call('Hello!')
    puts proc_2.call('This is Proc 2!!')
    puts proc_3.call('proc 3')
end

greeting_proc = Proc.new { | text | "#{text}, Michael!" }
repeat_proc = Proc.new { | text | text * 2 }
question_proc = Proc.new { | text | "Is this #{text} ?" }

hoge(greeting_proc, repeat_proc, question_proc)

#=> Hello!, Michael!
#   This is Proc 2!!This is Proc 2!!
#   Is this proc 3 ?
ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Rubyရဲ့‌ အခြေခံအချက်‌တွေ၊ Yield and Proc

Yield

block ကို run ဖို့အတွက် yield ကို အသုံးပြုနိုင်ပါတယ်။ ဒီနေရာမှာ block နဲ့ပတ်သက်ပြီး တစ်ခုပြောချင်ပါတယ်။block ဟာ do ～ end နဲ့｛｝ကြားထဲမှာ ရှိတဲ့ process တစ်ခုဖြစ်ပါတယ်။အောက်ပါ ဥပမာမှာဆိုရင် [puts i] ဟာ block ရဲ့ အစိတ်အပိုင်း တစ်ခုဖြစ်ပါတယ်။

(1..10).each do |i|
	puts i
end

လောလောဆယ် method ကိုခေါ်ပုံခေါ်နည်း 2 မျိုးရှိတယ်လို့မှတ်ထားပေးပါ။ပထမ ခေါ်ပုံခေါ်နည်းအနေနဲ့ method ကို ရိုးရိုးတန်းတန်းပဲ ခေါ်သုံးကြည့်ပါမယ်။ 

def hoge
    puts "hello hoge"
end

hoge
#=> hello hoge

နောက်တစ်နည်း အနေနဲ့ method ကို block ထည့်ပြီးတော့ ခေါ်ကြည့်လိုက်ပါမယ်။ 

def hoge
    puts "hello hoge"
end

hoge do
    puts "hello hoge2"
end
#=> hello hoge

အထက်ပါ code ကိုကြည့်ရင် method ကို block နဲ့ခေါ်သုံးလိုက်တဲ့အချိန်မှာ error တွေဘာတွေလည်းမတက်ပါဘူး။out put ကလည်း ဘာမှပြောင်းလဲမသွားပါဘူး။ဒါပေမယ့် block ထဲက code တွေကိုကျတော့လည်း run မသွားပါဘူး။အဲ့ဒီလိုအချိန်မှာ block ထဲက code တွေကို run သွားစေချင်ရင် yield ကို အသုံးပြုနိုင်ပါတယ်။ 

def hoge
    puts "hello hoge"
    yield
end

hoge do
    puts "hello hoge2"
end
#=> hello hoge
#   hello hoge2

မြင်တဲ့အတိုင်းပါပဲ block အတွင်းထဲက code တွေ run သွားပါပြီ။ yield ကို multiple ခေါ်မယ်ဆို block ကလည်း multiple အခေါ်ခံရပါတယ် ။

def hoge
  "Hoge1"
  yield
  yield
  "Hoge2"
end

hoge do
  puts "Hello"
end

#=> "Hoge1"
#=>	"Hello"
#=> "Hello"
#=> "Hoge2"

block သာရှိမနေဘူးဆိုရင် yield ကနေ block ကို လှမ်းခေါ်လိုက်တဲ့အချိန်မှာ error တက်ပါလိမ့်မယ်။

def hoge
  yield
end

hoge

#=> LocalJumpError (no block given (yield))

အဲ့ဒီလို error မတက်ရအောင် yield ကို run မယ့်အချိန်မှာ
block ပေးပြီးခေါ်တာလား ရိုးရိုးခေါ်တာလားဆိုတာကို စစ်လို့ရပါတယ်။
block ကနေ ပေးလိုက်တာလားဆိုတာကို block_given? method ကိုသုံးပြီး စစ်ကြည့်နိုင်ပါတယ်။

def hoge
  puts "hoge"
  #block ကိုပေးလိုက်တာလားဆိုတာကို စစ်တာ
  if block_given?
    yield
  end
end

hoge do
  puts "Hello"
end

#=> "hoge"
#=> "Hello"

#block မရှိဘဲ method ကိုခေါ်တာ
hoge
#=> "hoge" (အခုဆိုရင် error မတက်တော့ပါဘူး）

yield ကနေ block ကို parameter ပေးတာ။ block ရဲ့ return value ကို လက်ခံတာတွေ လုပ်နိုင်ပါတယ် ။ 

def hoge
  text = yield "Kentaro"
  puts text
end

hoge do |text|
  "hello, #{text}"
end

#=> hello, Kentaro

■block ကို parameter အနေနဲ့လက်ခံတာ 

yield ကိုမသုံးဘဲ နောက်တစ်နည်းအနေနဲ့ block ကို method ရဲ့ parameter အနေနဲ့ ပေးပြီးခေါ်သုံးလို့ရပါတယ်။
block ကို method ရဲ့ parameter အနေနဲ့ပေးမယ်ဆိုရင် parameter name အရှေ့မှာ [&] ကိုထည့်ပေးဖို့လိုပါတယ် ။
ပြီးရင် အဲ့ block ကို call method ကို သုံးပြီး run လို့ရပါတယ် ။

def hoge(&block)
  text = block.call('Kentaro')
  puts text
end

hoge do |text|
  "hello, #{text}"
end

#=> hello, Kentaro

block ကို parameter ပေးပြီးသုံးရတဲ့ ကောင်းကျိုးကတော့  block ကို တခြား method တွေဆီ ထပ်ပြီး parameter passing လုပ်လို့ရတာပဲ ဖြစ်ပါတယ်. 

def hoge(&block)
  introduction("Kentaro", &block)
end

def fuge(&block)
  introduction("Sakura", &block)
end

def introduction(text, &block)
  puts text
  puts block.call(text)
end

hoge do |text|
  "hello, #{text}"
end
#=> Kentaro
#=> hello, Kentaro

fuge do |text|
  "hello, #{text}"
end
#=> Sakura
#=> hello, Sakura

Rescue Modifier

rescue ကို modifier အနေနဲ့သုံးလို့ရပါတယ်။

exception တက်တဲ့အခါမှာ  return value က rescue ထဲကနေပြန်လာတဲ့ value ပဲဖြစ်မှာပါ

require 'date'

def to_date(string)
    begin
        Date.parse(string)
    rescue ArgumentError
        nil
    end
end

# exception မတက်ရင်
to_date('2021-05-19')
#=>  #<Date: 2021-05-19 ((2459354j,0s,0n),+0s,2299161j)>

# exception တက်ရင်
to_date('aaabbb')
#=> nil

အပေါ်က code ထဲက rescue modifier အပိုင်းကို အောက်ပါအတိုင်းအတိုချုံ့ရေးလို့ရပါတယ်. 

def to_date(string)
    Date.parse(string) rescue nil
end

# exception မတက်ရင်
to_date('2021-05-19')
#=>  #<Date: 2021-05-19 ((2459354j,0s,0n),+0s,2299161j)>

# exception တက်ရင်
to_date('aaabbb')
#=> nil

code ရဲ့ process ကအတူတူပါပဲချုံ့ရေးလို့ရတယ်လို့ဆိုလိုတာပါ.ဒါပေမဲ့ ဒီလိုချုံ့ရေးထားတဲ့အချိန်မှာတော့ ကိုယ်သတ်မှတ်ချင်တဲ့ exception class တွေကိုသတ်မှတ်လို့မရတာတော့ရှိပါတယ်.ဒီလိုမျိုးရေးရင် StandErrorနဲ့ သူ့ subclass တွေရဲ့ exception ကိုတက်ပေးမှာပါ.တကယ်လို့ ကိုယ်က exception class တွေနဲ့ အသေးစိတ်ရေးချင်တယ်ဆိုရင်တော့ begin～end ကိုသုံးပြီးရေးရမှာပါ.

ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Exception၊ Rubyရဲ့‌ အခြေခံအချက်‌တွေ

Exception ရဲ့ Return Value

Ruby ရဲ့ exception မှာ return value ရှိပါတယ်.return ပြန်တဲ့အ‌ခြေအနေက exception တက်တဲ့အချိန်နဲ့ exception မတက်တဲ့အချိန်ဆိုပြီး  ၂ မျိုး ၂ စားကွဲပါတယ်.exception မတက်တဲ့အချိန်ဆိုရင် begin အောက်က နောက်ဆုံး process ကို return ပြန်ပါတယ်.exception တက်တဲ့အချိန်ဆိုရင် rescue အောက်က နောက်ဆုံး process ကို return ပြန်ပါတယ်. 

# exception မတက်တဲ့အချိန်ဆိုရင် begin အောက်က နောက်ဆုံး process ကို return ပြန်ပါတယ်.
foo = 
    begin
        'Hello'
    rescue
        'error'
    ensure
        'ensure'
    end
foo #=> "Hello"
# exception တက်တဲ့အချိန်ဆိုရင် rescue အောက်က နောက်ဆုံး process ကို return ပြန်ပါတယ်.
bar = 
    begin
        1/0
    rescue
        'error'
    ensure
        'ensure'
    end
bar #=> "error"
ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Exception၊ Rubyရဲ့‌ အခြေခံအချက်‌တွေ

Exception ရဲ့ Else

Ruby ရဲ့ exception မှာ တစ်ခုရှိတာက exception မတက်တဲ့အချိန်မှာလုပ်ချင်တဲ့အလုပ်မျိုးကိုလုပ်ချင်တယ်ဆိုရင်else ကိုသုံးပြီးရေးလို့ရပါတယ်. 

begin
    # error တက်နိုင်မယ့် code
rescue
    # exception တက်မှ run မယ်
else
    # exception မတက်ရင် run မယ်
ensure
    # exception တက်တက်မတက်တက် run ကို run မှာ
end

အဲ့လိုရေးလို့ရတယ်ဆိုပေမယ့်လဲ တကယ့်တကယ့်သိပ်ရေးလေ့မရှိကြပါဘူး.exception မတက်ရင်က begin ～ rescue ကိုပဲ run တာဖြစ်လို့အဲ့ထဲမှာပဲ ရေးကြတာများပါတယ်

begin
    puts 'Hello World'
rescue
    puts 'exception တက်'
else
    puts 'exception မတက်ပါ'
end
#=> "Hello World"
#   "exception မတက်ပါ"

အထက်ပါ code နဲ့ output က အတူတူပဲဖြစ်မှာပါ 

begin
    puts 'Hello World'
    puts 'exception မတက်ပါ'
rescue
    puts 'exception တက်'
end
#=> "Hello World"
#   "exception မတက်ပါ"
ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Exception၊ Rubyရဲ့‌ အခြေခံအချက်‌တွေ

Retry and Ensure

တကယ်လို့ error တစ်ခုခုတက်ခဲ့ပြီး အဲ့ error ကို ထပ်ထပ်ပြီး ပြင်စေချင်တယ်ဆိုရင် rescue ထဲမှာ retry ကိုခံရေးလို့ရပါတယ်.retry ခံရေးထားလိုက်ရင် rescue ရဲ့ begin ကနေ အလုပ်ကို ပြန်စလုပ်ပေးမှာပါ.ဥပမာ network error တွေဘာတွေကြောင့် error တက်ခဲ့မယ်ဆိုရင် ထပ်ထပ်ပြီး retry လုပ်ချင်တယ်ဆိုတဲ့အခါမျိုးမှာသုံးလို့ရပါတယ်. 

begin
    # error တက်နိုင်မယ့် code
rescue
    retry # process ကို ပြန် run မယ်
end

အဲ့ဒီမှာ တစ်ခုသတိထားရမှာက process ကို ပြန်ပြန် run ပေးတဲ့အတွက် infinite loop ဖြစ်နိုင်တဲ့အလားအလာရှိပါတယ်.အဲ့လိုမဖြစ်အောင်ကတော့ count တွေ ဘာတွေနဲ့ စစ်ပေးလိုက်လို့ရပါတယ်. 

max_retry_count = 3
retry_count = 0
begin
    puts 'start process'
    1/0
rescue
    retry_count += 1
    if retry_count <= max_retry_count
        puts "#{retry_count} ခါ retry လုပ်မယ်"
        retry
    else
        puts 'retry လုပ်တာပြီးသွားပါပြီ'
    end
end

#=> start process
#   1 ခါ retry လုပ်မယ်
#   start process
#   2 ခါ retry လုပ်မယ်
#   start process
#   3 ခါ retry လုပ်မယ်
#   start process
#   retry လုပ်တာပြီးသွားပါပြီ

exception တက်သည်ဖြစ်စေ မတက်သည်ဖြစ်စေ code ကို run စေချင်တယ်ဆိုရင်တော့ ensure ကိုသုံးနိုင်ပါတယ်.ensure အောက်မှာထည့်ထားတဲ့ code ဟာ run ကို run သွားမယ့် code ဖြစ်ပါတယ်.   

begin
    # error တက်နိုင်မယ့် code
rescue
    # exception တက်မှ run မယ်
ensure
    # exception တက်တက်မတက်တက် run ကို run မှာ
end

ဥပမာ file ကို ဖွင့်တဲ့ process မျိုးဆိုရင် နောက်ဆုံးမှာ exception တက်တက်မတက်တက်file ကို ပိတ်ပေးဖို့လိုအပ်ပါတယ်အဲ့လိုမျိုးဆိုရင် ensure ကိုသုံးပြီး အောက်ပါအတိုင်းရေးနိုင်ပါတယ် 

# file ကိုဖွင့်မယ်
file = File.open('sampleFile.txt', 'w')
begin
    # file ထဲကို စာရိုက်ထည့်မယ်
    file << 'Hello World'
ensure
    # exception တက်ရင်လဲ file ကိုပိတ်မယ်.မတက်ရင်လဲ file ကိုပိတ်မယ်
    file.close
end

ensure အစား block ကိုသုံးရင်လဲရပါတယ်.
block ကိုသုံးပြီး file ကိုဖွင့်ရင် method ပြီးသွားတဲ့အချိန်မှာ file ကို auto ပိတ်ပေးမှာပါ. 

File.open('sampleFile.txt', 'w') do |file|
    file << 'Hello'
end
ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Exception၊ Rubyရဲ့‌ အခြေခံအချက်‌တွေ

Exception Class and Inheritance Relationship

rescue ထဲမှာက exception class ကိုကိုင်တွယ်ရမှာမို့ exception class ရဲ့ inheritance relationship တွေအကြောင်းကို သိထားဖို့လိုပါမယ်.ZeroDivisionError တို့ NoMethodError တို့လိုမျိုး exception class တွေအကုန်လုံးဟာ Exception Class ကြီးကို inheritance ယူထားတာပါ.Exception Class ကြီးရဲ့အောက်မှာမှ subclass လေးတွေအနေနဲ့ ထပ်ရှိနေကြတာပါ.Exception Class ကြီးရဲ့ subclass အနေနဲ့အောက်ပါ class တွေရှိကြပါတယ်ဆိုပါစို့.NoMemoryError, SystemExit…, StandardErrorအဲ့ဒီမှာ တစ်ခုသိထားရမှာက StandardError class အကြောင်းပါပဲ.error တက်နိုင်ခြေအများဆုံး exception တွေကို StandardError class ကနေလုပ်ပေးထားတာပါ.StandardError class အောက်မှာဆိုရင် RuntimeError, NameError, TypeError, ArgumentError,စသည်ဖြင့် error တက်နိုင်ခြေများတဲ့classများရှိကြပါတယ်.တကယ်လို့ rescue မှာ ဘာ class ကိုမှ သတ်မှတ်မပေးခဲ့ဘူးဆိုရင် StandardError class နဲ့ သူ့ရဲ့ subclass တွေရဲ့ exception တွေကိုပဲ run ‌ပေးမှာပါ.StandardError နဲ့ တစ်တန်းတည်းရှိတဲ့ NoMemoryError, SystemExit စတဲ့ Systemနဲ့ပတ်သက်တဲ့ exception တွေကို ဖမ်းပေးမှာမဟုတ်ပါဘူး. 

begin
    # error တက်နိုင်မယ့် code
rescue
    # StandardError class နဲ့ သူ့ရဲ့ subclass တွေပဲ exception တက်မယ်
end

အောက်ပါအတိုင်း Exception class ကို‌ ရေးပေးထားလိုက်ရင်တော့ exception class နဲ့ subclass တွေအကုန်လုံး(NoMemoryError, SystemExit အပါအဝင်, StandardError)ကို ဖမ်းပေးမှာပါ. 

begin
    # error တက်နိုင်မယ့် code
rescue Exception
    # Exception Class နဲ့သူ့ subclass အကုန်လုံး exception တက်မယ် (NoMemoryError, SystemExit..., StandardError)
end

ဒါပေမဲ့လို့ တကယ့်တကယ် code ရေးတဲ့အခါမှာ တိတိကျကျရေးဖို့လိုအပ်တာကြောင့် rescue မှာ Exception လို့ရေးလိုက်တာမျိုး..StandardError လို့ရေးလိုက်တာမျိုး မလုပ်သင့်ပါ.သူတို့ရဲ့ subclass တွေအထိ တိတိကျကျရေးပေးသင့်ပါတယ်.

subClass တွေအထိ တိတိကျကျရေးတဲ့နေရာမှာလည်း ဒီတစ်ချက်ကို သတိထားသင့်ပါတယ်.code ကို ကြည့်ပါ. 

begin
    # NoMethodError
    'foo'.bar
rescue NameError
    # NameError တက်တာကိုဖမ်းထားတာ
    puts 'NameError တက်နေပါတယ်'
rescue NoMethodError
    # NoMethodError တက်တာကိုဖမ်းထားတာ
    puts 'NoMethodError တက်နေပါတယ်'
end
#=> NameError တက်နေပါတယ်

အထက်ပါ code ကိုကြည့်ရင် NameError, NoMethodError တွေနဲ့ Exception ဖမ်းထားတာကို တွေ့ရမှာပါ.တက်တဲ့ error ကလည်း NoMethodError တက်တာပါ.ဒါပေမဲ့ NameError ကိုပဲ exception ဖမ်းသွားပါတယ်..ဘာကြောင့်ပါလဲ.. အဖြေက ဒီလိုပါ..အောက်ပါ class relationship တွေကို ကြည့်ပါ

Exception > StandardError > NameError > NoMethodError

NameError က NoMethodError ရဲ့ super class ဖြစ်နေတဲ့အတွက် ပထမဆုံးဖမ်းထားတဲ့ NameError exception ထဲဝင်သွားတာပါ.အဲ့လိုမျိုးမဖြစ်အောင် ဘယ်လိုဖြေရှင်းရမလဲဆိုတာကတော့ လွယ်ပါတယ်.subclass ဖြစ်တဲ့ NoMethodError exception ကို အရင်ဖမ်းထားလိုက်ပါပြီးမှ သူ့ရဲ့ super class တွေကို exception ဖမ်းပါ. 

begin
    # NoMethodError
    'foo'.bar
rescue NoMethodError
    # NoMethodError တက်တာကိုဖမ်းထားတာ
    puts 'NoMethodError တက်နေပါတယ်'
rescue NameError
    # NameError တက်တာကိုဖမ်းထားတာ
    puts 'NameError တက်နေပါတယ်'
end
#=> NoMethodError တက်နေပါတယ်

NoMethodErrorတက်တာကတော့ မှန်သွားပါပြီNameErrorတက်တာကကော မှန်လား ကြည့်ကြည့်ပါမယ်.အောက်ပါအတိုင်း NameError လည်း မှန်မှန်ကန်ကန် exception ဖမ်းပေးပါတယ်. 

begin
    # NameError
    Foo.new
rescue NoMethodError
    # NoMethodError တက်တာကိုဖမ်းထားတာ
    puts 'NoMethodError တက်နေပါတယ်'
rescue NameError
    # NameError တက်တာကိုဖမ်းထားတာ
    puts 'NameError တက်နေပါတယ်'
end
#=> NameError တက်နေပါတယ်
ရေးသား ထုတ်ဝေခဲ့ ချိန် စာရေးသူ ကဏ္ဍများ Exception၊ Rubyရဲ့‌ အခြေခံအချက်‌တွေ

Creating Execption

Execption တစ်ခုကိုအောက်ပါအတိုင်း create လုပ်နိုင်ပါတယ်. 

begin 
    # error တက်နိုင်မယ် code ကို ဒီအထဲမှာရေး
rescue => e # exception object ရဲ့ variable
    # begin ～ rescueကြားထဲမှာ error တက်ရင် လုပ်မယ့် code ကို ဒီမှာ‌ရေး
end

e ဆိုတာက exception object ကိုထည့်ထားတဲ့ variable တစ်ခုပါ..error သာ တက်ခဲ့မယ်ဆိုရင် အဲ့ e variable ထဲကနေ data တွေကို ထုတ်ယူပြီးကြည့်နိုင်ပါတယ်.တစ်ချို့တွေကျတော့ e အစား ex ဆိုတဲ့နာမည်ကိုလည်းပေးတတ်ကြပါတယ်.e.class ဆိုပြီး class ကိုခေါ်ကြည့်ရင် ဘယ် class ရဲ့ exception မှာ error တက်တာလဲဆိုတာကိုသိရမှာဖြစ်ပါတယ်.message method ကိုခေါ်ကြည့်ရင် error message ကို သိရမှာဖြစ်ပြီးတော့backtrace method ကို ခေါ်ကြည့်ရင် ခေါ်ခဲ့တဲ့ method history တွေကို သိရမှာဖြစ်ပါတယ်. 

begin 
    # error တက်လာနိုင်မယ့် code 
    1 / 0
rescue => e # exception object ရဲ့ variable
    # begin~rescueကြားထဲမှာ error တက်ရင် လုပ်မယ့် code ကို ဒီမှာ‌ရေး
    puts "error class: #{e.class}"
    puts "error message: #{e.message}"
    puts "backtrace......."
    puts e.bakctrace
    puts "................."
end

#အထက်ပါ code ကို run ရင် အောက်ပါအတိုင်း ထွက်လာပါလိမ့်မယ်.
#=> error class: ZeroDivisionError
#   error message: divided by 0
#   backtrace.......
#   (irb):11:in `/'
#   (irb):11:in `rescue in irb_binding'
#   (irb):10:in `irb_binding'

#   C:/Ruby26-x64/bin/irb.cmd:31:in `<main>'
#   .................

ကိုယ်တွေမှာ error တက်နိုင်မယ့် code တစ်ခုရှိတယ်ဆိုကြပါစို့ .. ဥပမာ 

begin 
    1 / 0
end

အထက်ပါ code ကို run လိုက်ရင် အောက်ပါအတိုင်း ZeroDivisionError တက်ပါလိမ့်မယ်.

ZeroDivisionError (divided by 0)

အဲ့သလို error တက်လာခဲ့ရင် ဘာလုပ်မလဲဆိုတာမျိုးကို rescue အောက်မှာရေးကြရမှာပါ. 

begin 
    1 / 0
rescue => e
    puts "0 နဲ့စားလို့ error တက်သည်"
end
#=> 0 နဲ့စားလို့ error တက်သည်
